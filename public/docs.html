Vyper Game Editor

A browser-based 3D & 2D game editor that pairs a Babylon.js engine with a React UI — author entity behaviour with a visual node graph or hand-written JavaScript, and switch between them at any time. With rigid-body physics, particle effects, a HUD editor, one-click players, and a goals system that ties it all to your game's objectives.

This guide documents everything the editor can do today, with diagrams and step-by-step how-tos. Everything runs entirely in your browser — no build step, no accounts, no backend.

Install & run

# install dependencies
npm install

# start the dev server → http://localhost:5173
npm run dev

# type-check + production bundle
npm run build

# headless smoke test of codegen + runtime compile
npx tsx scripts/smoke.ts

The editor opens with a starter scene: a Sun (directional light), a Ground plane, and a Player box. You can start editing immediately.

2D & 3D games

When you create a game on the Home screen, you pick its type — 3D or 2D. The choice is saved with the game and shown as a badge on its card. The same editor and workflow are used either way; only the camera, rendering and tooling auto-configure to match.

Aspect3D2D
Editor cameraOrbit (perspective) — drag to rotate, wheel to zoom.Orthographic, head-on. Orbit is locked; wheel zooms, Ctrl/middle-drag pans.
Game cameraFree perspective rig you position anywhere.Orthographic frame that pans in X/Y at a fixed depth.
Shapesbox · sphere · cylinder · cone · plane · groundsquare · circle · triangle · plane
LightingLights + lit materials.Flat, unlit fills — no lights required.
GridOn the XZ floor.On the XY plane, facing you.
GizmosFull X/Y/Z.Constrained to the plane (no Z-move, Z-only rotate, no Z-scale).
Scripting is identical in both modes — entities still have position, rotation, props and the same API. In 2D you simply work in X/Y (Z is depth/layering).

The workspace

Vyper is a fully resizable six-pane workspace. Drag any divider to resize; layouts persist between sessions.

Hierarchy — scene tree. Add, duplicate, delete and select entities; toggle the grid; reach the Game Camera.
Scene viewport — the editor camera with a grid, click-to-select picking and transform gizmos.
Game preview — the same scene through the play camera, with an inline Play button.
Script editor — per-entity behaviours, switchable between Nodes and Code.
Debugger — live console, level filters, repeat-collapsing and FPS / mesh counters.
Inspector — transform, mesh/light props, custom properties and attached behaviours; live while playing.

Make your first scene

  1. Add an object. Click Mesh + in the toolbar and pick box (or right-click the viewport → Add Mesh). It appears at the origin and is auto-selected.
  2. Position it. Press the Move tool, drag the gizmo arrows in the scene view, or type exact values into the Inspector's Transform fields.
  3. Give it behaviour. In the Inspector, click + Add Behaviour. A starter node graph is attached that logs on start and spins the object on update.
  4. Play. Hit ▶ (or the Play button on the Game preview). The box spins; the Debugger prints “Entity spawned”.
  5. Iterate. Press Stop to return to the exact pre-play state, tweak the graph or code, and play again.

Adding objects

Add objects from the toolbar (Mesh + / Shape + · Player + · Light + · FX +), by right-clicking the viewport, or by right-clicking empty space in the Hierarchy. The menus adapt to the game's type.

3D primitives

box · sphere · cylinder · cone · plane · ground · empty

Each mesh gets a cycling neon color and is placed at the origin (flush to the floor for ground/plane). empty is a transform-only node with no geometry — useful as a parent or marker.

2D shapes

square · circle · triangle · plane · empty

Flat shapes lying in the XY plane, rendered unlit so they’re visible without any lights. Placed at the origin on the z=0 plane.

Player — ready to drive

One click drops a player object with default movement controls already attached. Hit Play and go.

2D players move on the XY plane with WASD/arrows; 3D players get a mouse-look third-person controller (move + look + jump). Tune speeds on the controller node, or swap in a first-person controller. See behaviours and the controller nodes.

Lights (3D only)

hemispheric · point · directional

Each light has an editable color and intensity. The 3D starter scene ships with a directional “Sun”. 2D games skip lighting entirely.

FX — particle effects

With an object selected, FX + attaches a particle effect from a preset (fire, smoke, explosion, sparkle…).

Tune it in the Inspector or trigger it from scripts. See Effects (VFX).

Tip: Duplicate the selected object with ⌘D (Maya/Unity) or copy/paste it — attached behaviours and effects are copied too, with fresh ids.

Transform & gizmos

Four transform tools live in the toolbar. The active one drives the on-screen gizmo for the selected object.

Click an object in the viewport to select it; the gizmo snaps to it. Drag a gizmo handle to transform, or type precise numbers into the Inspector. Press the focus key (F in Maya/Unity, . in Blender) to frame the selection.

Game camera & preview

Vyper renders the scene twice at once (Babylon multi-view): the Scene panel uses a free editor camera you can orbit; the Game panel uses the dedicated play camera — exactly what a player would see.

  • Select Game Camera in the Hierarchy's Editor group.
  • Edit its Position and Rotation in the Inspector, or move its rig in the Scene view.
  • Use Reset to default to return to the standard behind-and-above framing.
The grid is an editor-only aid — it never appears in the Game preview or while playing.

Physics

Vyper has a built-in rigid-body simulation (Havok). Give any mesh a body in the Inspector's Physics section, then objects fall, collide and bounce while playing. Physics runs in 3D games; 2D movement is script-driven.

SettingWhat it does
Rigid bodyMaster toggle — off means no body (the mesh is inert / decorative).
Typedynamic (fully simulated, affected by gravity) · static (never moves — walls/floors) · kinematic (moved by script, pushes dynamics).
ShapeCollider: auto (picked from the mesh kind) · box · sphere · capsule · cylinder.
MassKilograms (dynamic only). Ignored for static bodies.
BounceRestitution 0–1 — how much energy is kept on impact.
FrictionSurface friction 0–1.
Floors are automatic: on Play, any ground or plane mesh gets a static collider even without an explicit body, so players and dynamic objects have something to stand on.

Physics in scripts & nodes

The Physics node category and the matching script API drive bodies at runtime. A behaviour can even create its body on the fly with entity.usePhysics() (this is how the player controllers work).

Effects (VFX)

Attach particle effects to any object for fire, smoke, explosions, magic and weather. Select an object and click FX + in the toolbar (or use the Inspector's Effects section) to add one from a preset. Effects are drawn by Babylon's particle system and persist with the scene.

Presets

Fire · Smoke · Explosion · Sparkle / Magic · Confetti · Muzzle Flash · Heal / Aura · Rain · Snow · Dust Trail · Electricity

The menu lists only presets that suit the current 2D/3D mode. Each one is a fully editable starting point — not a black box.

The effect editor

Open an effect from the Inspector to tune emitter shape, emit rate, particle size/lifetime, color-over-life gradients, gravity, blend mode and texture — with a live preview.

Playback & triggering

  • Auto playback starts the effect at Play (optionally looping, with a duration and delay).
  • Manual playback waits to be triggered — from a script (entity.playEffect()) or the FX nodes.
NodeKindEffect
Play EffectactionStarts an effect on this object (by name, or all effects if blank).
Stop EffectsactionStops this object's emitting effects.
Effects are stored on the entity, so copy/paste and duplicate bring them along.

HUD editor

Click HUD in the toolbar to open the on-screen overlay editor — the same tool for both 2D and 3D games. Design health bars, score, timers, a crosshair, objectives and more on a 16:9 stage; it renders live over the Game preview so you see exactly how it sits in-game, both while editing and during play.

  • Add a widget by clicking it in the palette. It drops onto the stage selected.
  • Place it by dragging, or grab a handle to resize. The 3×3 anchor buttons snap it to a screen region. Everything is stored as a % of the screen, so it scales to any resolution.
  • Bind live values — give a bar/score/ammo widget an Object (e.g. Player) and a Prop (e.g. health). While playing, it reads that entity's live property; otherwise it shows the static preview value.
  • Style color, background, font, corner radius, opacity and alignment in the inspector.
The HUD is saved per game and shared across all its scenes. It draws over the Game preview at all times, so you're always seeing the real in-game result.

Goals & objectives

Click Design in the toolbar to open the game-design editor — a guided space for the rules and goals that turn a scene into a game. What you define here is saved with the game, shared across scenes, and wired into the node editor and runtime so objects can act in line with the game's goals.

What you author

  • Concept — a one-line pitch.
  • Win / Lose conditions — in plain language.
  • Objectives — the trackable goals.
  • Rules — the constraints of play.

A live completeness checklist nudges you toward a fully thought-out design.

An objective has

  • Priority — primary (required to win), secondary, or bonus.
  • Tracking — a flag (done / not done) or a counter toward a target (e.g. collect 5).
  • Reward — an optional note on what completing it does.

Using goals in behaviours

Every objective shows up in the node editor. Drop an Objective node and pick a goal from the dropdown — so any object can complete a goal, add progress, or branch on whether it's done.

NodeKindEffect
Complete ObjectiveactionMarks the chosen goal complete.
Add ProgressactionAdds an amount to a counter goal (completes it at the target).
Objective Complete?value → boolWhether the goal is done — wire into a Branch to gate behaviour.
Objective Progressvalue → numberThe goal's current count.

The same actions are available in code through the world API (see Scripting API). While playing, completing a goal logs to the Debugger, and once every primary objective is done Vyper announces the win.

Objectives are also a HUD widget — drop an Objective widget in the HUD editor to show progress on screen.

Behaviours overview

A behaviour (Script) is attached to an entity and defines what it does at runtime. Every behaviour holds both a node graph and JavaScript, kept in sync bi-directionally. Author visually, in code, or move fluidly between the two — see Node ↔ Code.

Nodes mode visual

Drag and wire nodes. Every edit regenerates the JS so the Code tab always mirrors the graph.

Code mode javascript

Hand-edit JS with full Monaco autocomplete. Each change is parsed straight back into the node graph, so the Nodes tab mirrors your code too.

Multiple behaviours can be attached to one entity, each individually enabled/disabled. They show as tabs across the top of the Script editor.

Visual node graph

The node editor is a React-Flow canvas. A palette on the left lists every node, grouped by category. Click a palette item to drop it in, or drag it onto the canvas. Wire nodes together to define flow and data.

Connections are validated as you draw them: a data port only accepts the same data kind, and a target input takes a single connection (the newest wins). An exec output can fan out to several targets — they run in top-to-bottom order — so one event can drive multiple action chains without a relay node.

Controllers — one node = a full behaviour

Plug-and-play macros that inject complete movement logic. Add one to an empty behaviour (or use the Player + button, which sets one up for you).

NodeUse
2D Player ControllerWASD / arrow movement on the XY plane (2D games).
First-Person ControllerMouse-look + WASD + jump, eyes-in-head camera (3D).

Controller nodes expose their tuning (move/sprint speed, jump height, look sensitivity, camera distance) as fields. Flip the behaviour to Code to read and edit the real controller logic.

Node ↔ Code round-trip

The graph and the code are two views of one behaviour, kept in sync both ways. Edit the nodes and the code updates; edit the code and the graph rebuilds to match. Neither is a black box, and neither silently wins.

Example — this graph and this code are the same behaviour:

// ⚙ On Start → Log "Entity spawned";  On Update → Rotate 60°/s on Y
function onStart(dt) {
  console.log("Entity spawned");
}

function onUpdate(dt) {
  { const _v = vec(0, 60, 0); entity.rotate(_v.x * dt, _v.y * dt, _v.z * dt); }
}

Change the 60 to 120 in the Code tab and the Rotate node's field updates to 120 the moment you flip to Nodes — and vice-versa. The Code tab shows a ⇄ synced badge while the graph mirrors your code.

When code can't be drawn as nodes

The node vocabulary covers the supported actions and values (see the catalog above). If you hand-write something it can't represent — a for loop, a function call, a comparison — Vyper doesn't mangle the graph. It pauses the sync, keeps your code as the source of truth, and shows a ⚠ can't be represented as nodes notice.

Reset from nodes: while the graph is paused, this button (Code tab) discards the unrepresentable code and rebuilds it from the last good graph — the explicit escape hatch back into sync.

Scripting API

Whether generated or hand-written, every behaviour runs with the same injected globals. Define onStart(dt) and/or onUpdate(dt); both are optional.

// available in every behaviour
entity   // transform + props + physics + effects (see table)
input    // { key(name), axisX, axisY, mouse, lockPointer() }
time     // { elapsed, delta }            — seconds
vec(x, y, z) // build a {x,y,z} (with .add/.scale/.normalize)
camera   // game camera: yaw/pitch, follow helpers
world    // other objects + game objectives
scene    // the raw Babylon Scene (advanced)
console  // log / info / warn / error / debug → Debugger panel

function onStart(dt) { /* once, at Play */ }
function onUpdate(dt) { /* every frame; dt = seconds since last frame */ }
function onCollision(other) { /* when a physics body touches another */ }

Example — WASD movement

function onUpdate(dt) {
  const speed = entity.props.speed ?? 3;
  entity.translate(input.axisX * speed * dt, 0, input.axisY * speed * dt);
}
Crash safety: compile and runtime errors are caught and printed to the Debugger. A behaviour that throws is disabled for the rest of the session so it can't spam the console at 60 fps. In Nodes mode the failing node is pinpointed in the live flow.

Play / Pause / Stop

The transport controls in the toolbar drive the runtime. The state pill shows editing · playing · paused.

▶ Play

Snapshots the scene, compiles every enabled behaviour, runs onStart, then drives onUpdate(dt) from the render loop.

❚❚ Pause

Freezes updates; press again to resume from exactly where you stopped.

■ Stop

Restores the pre-play snapshot non-destructively — your scene returns to its exact edit state.

While playing, the Inspector shows live transforms and properties, and the Game preview is the canonical view. Editing is locked to prevent fighting the running simulation.

Debugger console

The bottom panel captures all console.* output from behaviours and surfaces live engine stats.

  • Level filters — toggle log / info / warn / error / debug independently.
  • Collapse repeats — identical consecutive lines fold into one row with a count badge (great for per-frame logs).
  • Live stats — current FPS (highlighted while playing) and total mesh count.
  • Source tags — each line is labelled with the behaviour that emitted it.
  • Clear — wipe the log.

Babylon Inspector

Click Inspector in the toolbar to overlay Babylon.js's own debug layer — a deep scene inspector for nodes, materials, textures, the render pipeline and live property tweaking. It loads on demand and toggles on/off without disturbing your scene.

This is distinct from Vyper's own right-hand Inspector panel. Use it when you need engine-level detail beyond what the editor surfaces.

Undo, redo & clipboard

Vyper keeps a snapshot-based history (up to 80 steps) of the editable scene — entities, behaviours and selection. Rapid same-kind edits (dragging a field, typing code) coalesce into a single undo step so one drag isn't 200 undos.

  • Undo / Redo from the toolbar arrows or the keymap bindings above.
  • Copy / Paste an entity — its behaviours come along as fresh copies, offset slightly so it doesn't overlap the original.
  • Camera & grid are view state, deliberately excluded from undo history.
